# Инструкции для Claude CLI — AutoCooker Bug Fix Agent

> Этот файл описывает твой рабочий процесс. Прочитай его целиком перед любыми действиями.
> Все пути — относительно корня проекта AutoCooker, если не указано иное.
> Сопроводительные файлы: `AUTOCOOKER_FIXES.md` (очередь) и `FIXED_BUGS.md` (архив).

---

## Часть 1 — Стартовый чеклист (выполнять каждый раз при запуске)

```
[ ] 1. Прочитать AGENT_INSTRUCTIONS.md (этот файл) целиком
[ ] 2. Прочитать AUTOCOOKER_FIXES.md — определить, есть ли неисправленные баги
[ ] 3. Если FIXED_BUGS.md существует — прочитать его, чтобы не повторять уже сделанное
[ ] 4. Выбрать следующий баг по приоритету (см. Часть 2)
[ ] 5. Выполнить исправление (см. Часть 3)
[ ] 6. Обновить AUTOCOOKER_FIXES.md и FIXED_BUGS.md (см. Часть 4)
[ ] 7. Если багов в AUTOCOOKER_FIXES.md не осталось — перейти к Части 5
```

## Часть 2 — Приоритет выбора следующего бага

Читай список из `AUTOCOOKER_FIXES.md` и выбирай по следующей шкале:

```
1. 🔴 КРИТИЧЕСКИЙ  (FIX-1, FIX-2, SIMP-1, SIMP-3)
   — блокирует работу или вызывает 100% failure
   — брать в первую очередь, даже если сложный

2. 🟠 ВЫСОКИЙ      (FIX-3, FIX-4, FIX-5, FIX-6, SIMP-2, SIMP-4, SIMP-7)
   — значительно деградирует качество или скорость
   — брать сразу после критических

3. 🟡 СРЕДНИЙ      (FIX-7, FIX-8, FIX-9, SIMP-5, SIMP-6)
   — улучшение без блокировки
   — брать после высоких

4. ⚪ НИЗКИЙ       (FIX-10, SIMP-8, SIMP-9)
   — косметика, оптимизация
   — брать последними или пропускать если нет времени
```

**Правило зависимостей:** некоторые баги зависят друг от друга. Перед началом проверяй:
- FIX-2 требует FIX-1 (нет смысла менять tools без правильного executor)
- FIX-8 требует FIX-7 (done_summary нужен для completed_summary)
- SIMP-6 требует понимания SIMP-1 (оба про cache в system prompt)
- FIX-10 делать только после FIX-1 + FIX-2 + FIX-9

## Часть 3 — Процедура исправления одного бага

### 3.1 Подготовка

```
1. Открыть раздел бага в AUTOCOOKER_FIXES.md
2. Прочитать: "Файл", "Проблема", "Строки", "Исправление"
3. Найти указанные строки в реальном файле:
   - НЕ доверяй номерам строк из документа — они могли сместиться
   - Используй grep или поиск по уникальной строке из "Что искать"
4. Прочитать контекст вокруг (±30 строк) чтобы понять структуру
```

### 3.2 Выполнение

```
1. Применить изменение точно как описано в "Исправление"
2. Если описание неоднозначно — применить минимально инвазивный вариант
3. Не делать попутных рефакторингов
4. Не переименовывать переменные если это не часть фикса
5. Сохранить файл
```

### 3.3 Верификация после изменения

После каждого фикса выполнить соответствующую проверку:

| Тип фикса | Как проверить |
|---|---|
| Изменение пути executor | `grep -n "_make_executor" <файл>` — убедиться что нужный вызов изменён |
| Новая константа/переменная | `grep -n "НОВОЕ_ИМЯ" <файл>` — убедиться что добавлена и используется |
| Изменение числового параметра | `grep -n "старое_число" <файл>` — убедиться что не осталось дублей |
| Изменение промпта | Прочитать файл целиком и убедиться в логике |
| Добавление новой функции | Убедиться что функция вызывается в нужном месте |

### 3.4 Smoke test (если проект запускаем)

Если проект доступен для запуска — после каждого критического фикса:
```bash
cd /path/to/AutoCooker
python -c "from core.phases.coding import CodingPhase; print('Import OK')"
python -c "from core.phases.planning import PlanningPhase; print('Import OK')"
python -c "from core.tools import CRITIC_SUBPHASE_TOOLS; print('Tools OK')"
```

## Часть 4 — Обновление файлов после исправления

### 4.1 Добавить в FIXED_BUGS.md

Создай файл если не существует. Формат записи:

```markdown
## [FIX-N] Краткое название бага

**Файл(ы):** `путь/к/файлу.py`
**Дата:** YYYY-MM-DD
**Приоритет:** 🔴 Критический / 🟠 Высокий / 🟡 Средний / ⚪ Низкий

### Что было
Одно-два предложения: в чём состояла проблема.

### Что сделано
Конкретно: какие строки изменены, что добавлено/удалено.

### Diff (ключевые строки)
```python
# ДО:
старый_код

# ПОСЛЕ:
новый_код
```

### Проверка
Как убедился что работает.
```

### 4.2 Убрать из AUTOCOOKER_FIXES.md

После добавления в FIXED_BUGS.md — удалить соответствующий раздел из AUTOCOOKER_FIXES.md.

**Не удалять:**
- Заголовок документа
- Секцию "Контекст: как работает пайплайн"
- Сводную таблицу (она нужна для навигации, просто пометить строку ✅)
- Секцию "Ожидаемый результат"

**Удалять:**
- Весь раздел от `## FIX-N` до следующего `## FIX-` или `## SIMP-`

## Часть 5 — Самостоятельное исследование проекта

Выполнять когда `AUTOCOOKER_FIXES.md` пуст или содержит только LOW-приоритетные баги.

### 5.1 Структура исследования

```
Шаг 1: Прочитать ключевые файлы
  - core/phases/coding.py   (главная логика кодинга)
  - core/phases/planning.py (планирование)
  - core/phases/base.py     (run_loop, build_system)
  - core/ollama_client.py   (chat_with_tools)
  - core/tools.py           (tool definitions)
  - core/state.py           (KanbanTask, FileCache)

Шаг 2: Прочитать logs.json любой провальной задачи из .tasks/
  - Найти паттерны: что повторяется? что блокирует?
  - Подсчитать: сколько tool calls, сколько JSON INVALID, сколько Path escape

Шаг 3: Проанализировать промпты в prompts/
  - Есть ли противоречия между промптами?
  - Есть ли инструкции которые модель игнорирует?
  - Есть ли места где модель стабильно галлюцинирует?

Шаг 4: Проверить новые потенциальные проблемы (список ниже)
```

### 5.2 Чеклист для самостоятельного поиска багов

Проверять по порядку — от наиболее вероятных к редким:

```
ПРОИЗВОДИТЕЛЬНОСТЬ
[ ] Есть ли N+1 проблемы: вызов ollama.complete() в цикле по файлам?
[ ] Есть ли файлы которые читаются > 3 раз в одном run_loop?
[ ] Есть ли места где messages растёт без ограничений?
[ ] Есть ли run_loop с max_tool_rounds > 20 без обоснования?

ГАЛЛЮЦИНАЦИИ
[ ] Есть ли промпты длиннее 5KB без структурированного output-примера?
[ ] Есть ли места где модель получает противоречивые инструкции
    (в system AND в user message)?
[ ] Есть ли validate_fn которые не проверяют реальное содержимое файла,
    только его существование?
[ ] Есть ли subtask-промпты где "description" позволяет широкую интерпретацию?

НАДЁЖНОСТЬ
[ ] Есть ли необработанные исключения в tool handlers?
[ ] Есть ли места где path joining не нормализует слэши (Windows vs Linux)?
[ ] Есть ли file operations без encoding="utf-8"?
[ ] Есть ли json.load() без try/except?
[ ] Есть ли места где validate_fn вызывается до того как файл мог быть записан?

АРХИТЕКТУРА
[ ] Есть ли дублирование логики между coding.py и planning.py?
[ ] Есть ли магические числа (42, 1500, 8000) без именованных констант?
[ ] Есть ли функции длиннее 100 строк которые можно разбить?
[ ] Есть ли dead code (функции/методы которые никогда не вызываются)?
```

### 5.3 Критерии для добавления нового бага

Добавлять в `AUTOCOOKER_FIXES.md` только если:

```
🔴 КРИТИЧЕСКИЙ  — вызывает 100% failure или data loss
                  ИЛИ блокирует основной пайплайн
                  ИЛИ приводит к бесконечному циклу

🟠 ВЫСОКИЙ      — увеличивает время выполнения > 2× для типичных задач
                  ИЛИ вызывает failure в > 30% задач
                  ИЛИ вызывает систематические галлюцинации

🟡 СРЕДНИЙ      — ухудшает качество результата
                  ИЛИ увеличивает время на 20–100%
                  ИЛИ создаёт мусор в логах/файлах

НЕ добавлять:
  - субъективные предпочтения по стилю кода
  - оптимизации без измеримого эффекта
  - изменения "на всякий случай"
```

### 5.4 Формат нового бага в AUTOCOOKER_FIXES.md

```markdown
## NEW-N — Короткое название

### Файл
`путь/к/файлу.py`

### Проблема
Конкретное описание: что именно происходит, почему это плохо.
Если возможно — пример из логов или конкретные строки кода.

### Строки
`файл.py`, строки X–Y:
```python
# проблемный код
```

### Исправление
```python
# исправленный код
```

### Приоритет
🔴 / 🟠 / 🟡 / ⚪ — обоснование
```

## Часть 6 — Правила поведения при неопределённости

### Если описание фикса неоднозначно
```
1. Прочитать весь метод/функцию целиком (не только указанные строки)
2. Прочитать вызывающий код (где метод используется)
3. Применить наименее рискованный вариант
4. Задокументировать в FIXED_BUGS.md что именно выбрал и почему
5. НЕ угадывать — если совсем непонятно, пропустить и взять следующий баг
```

### Если фикс ломает что-то другое
```
1. Откатить изменение
2. Задокументировать проблему в FIXED_BUGS.md как "SKIPPED: причина"
3. Добавить в AUTOCOOKER_FIXES.md уточнение к этому багу
4. Перейти к следующему по приоритету
```

### Если найден баг который не описан в AUTOCOOKER_FIXES.md
```
1. Если критический — исправить сразу, задокументировать в FIXED_BUGS.md
2. Если средний/низкий — добавить в AUTOCOOKER_FIXES.md и продолжить текущую задачу
3. НЕ делать "попутных" правок без документирования
```

### Если проект не запускается после фикса
```
1. python -c "import core.phases.coding" — найти SyntaxError
2. python -c "import core.tools" — найти ImportError
3. Откатить последнее изменение
4. Перечитать раздел фикса ещё раз
5. Применить более осторожный вариант
```

Когда нужно быстро найти место для фикса:

| max_critique_iterations | `grep -n "max_critique_iterations" core/phases/planning.py` |

| critic_feedback сброс | `grep -n "critic_feedback = \[\]" core/phases/coding.py` |

| DEBUG логи | `grep -rn "DEBUG\]\|debug\]" core/phases/` |
