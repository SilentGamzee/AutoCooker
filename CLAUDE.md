```bash
pip install -r requirements.txt
python main.py
```

Requires Python 3.10+, Chrome/Chromium, and Ollama on `http://localhost:11434` (or LM Studio on `http://localhost:1234`). At least one model must be pulled: `ollama pull llama3.1`.

Check imports after changes:
```bash

```

## Architecture

AutoCooker is a Kanban AI coding assistant. The user creates a task in the browser UI → the backend runs a 3-phase pipeline → result lands in "Done" or "Human Review".

### Pipeline

```
Planning → Coding → QA
```

**Planning** (`core/phases/planning.py`, ~125K): Discovery → Requirements → Spec → Critique loop (3 sub-phases: scope/symbols/simplicity) → Implementation Plan (`implementation_plan.json`). Outputs structured subtasks.

**Coding** (`core/phases/coding.py`): Executes each subtask from the plan — passes implementation steps to LLM, runs coding critic sub-phases, writes files to isolated `_workdir/`, then generates README.

**QA** (`core/phases/qa.py`): Read-only. Runs tests, reviewer loop (up to 4 iterations), returns verdict. Has **no file-write tools**.

### Key Files

| File | Role |
|------|------|
| `main.py` | Eel entry point, all `@eel.expose` API functions, pipeline orchestrator |
| `core/phases/base.py` | `BasePhase` — `run_loop`, prompt loading, gevent-safe logging |
| `core/state.py` | `KanbanTask`, `Subtask`, `AppState`, `LogEntry` dataclasses |
| `core/tools.py` | Tool schemas (OpenAI function-call format) + `ToolExecutor` |
| `core/ollama_client.py` | API client for Ollama/LM Studio, tool invocation, session management |
| `core/sandbox.py` | Isolated workdir per task; all file ops are path-validated against it |
| `core/critic.py` | `RuleCritic` — rules-based feedback used by both Planning and Coding critique loops |
| `core/project_index.py` | Scans and caches project files for context injection |
| `prompts/` | 18 Markdown prompt files loaded at runtime via `load_prompt(filename)` |
| `web/js/app.js` | All frontend logic (~41K); communicates with backend via `eel.*()` calls |

### Data Model

```python
KanbanTask
  └── subtasks: list[Subtask]       # generated by Planning, executed by Coding
  └── logs: list[LogEntry]          # persisted to .tasks/task_NNN/logs.json
  └── models: dict[str, str]        # {"planning": "...", "coding": "...", "qa": "..."}
  └── column: str                   # Kanban column name
  └── phase_status: dict[str, str]  # per-phase progress

Subtask
  └── id: str                       # T-001, T-002, ...
  └── files_to_create/modify        # validated against project files
  └── implementation_steps          # list of {action, code, find, replace, ...}
```

### Task Storage

Each task gets an isolated directory:
```
.tasks/
└── task_021/
    ├── _workdir/           ← project copy; LLM writes here
    ├── logs.json
    ├── implementation_plan.json
    ├── spec.json
    └── context.json
```

### Tool Sets

Three tool sets (defined in `core/tools.py`) are swapped in per phase:
- `PLANNING_TOOLS` — read + write for spec/plan files
- `CODING_TOOLS` — adds `run_shell`, `confirm_task_done`, `create_task`
- `QA_REVIEWER_TOOLS` — read-only + `test_runner`

### Critique Loop Pattern

Used in both Planning and Coding phases:
```python
for iteration in range(max_iterations):
    output = run_subphase(prompt, tools)
    feedback = critic.evaluate(output)
    if feedback["passed"]:
        break
    # inject feedback and retry
```
Sub-phases A/B/C (`p4a`/`p4b`/`p4c` for Planning, `p6a`/`p6b`/`p6c` for Coding) each write their own JSON critique file that the validator checks.

### Gevent / Threading

`BasePhase.run()` runs in a background thread. All `eel.*` calls must go through:
```python
self._gevent_safe(lambda: eel.task_log_added(task_id, entry))
```
Never call `eel.*` directly from phase code.

## Prompts

All prompts are in `prompts/*.md` and loaded at runtime. Rules for editing them:

- **Do not hardcode project-specific paths** (e.g. `web/js/app.js`, `core/state.py`) — use generic descriptions like "the main JS application file" or "the state/dataclass file". Exception: stable architectural names like `main.py` are acceptable.
- `p5_impl_plan.md` is the most critical and most frequently failing prompt — changes here affect first-round correctness.
- Prompt changes are tracked in `AGENT_INSTRUCTIONS.md` (bug fix workflow) and `AUTOCOOKER_FIXES.md` (priority queue).

## Development Notes

- **`AGENT_INSTRUCTIONS.md`** — Russian-language guide covering the full bug-fix workflow, fix format (NEW-N entries), and rules for prompt changes. Read this before making systemic changes.
- **`AUTOCOOKER_FIXES.md`** — Active bug queue (🔴 critical → ⚪ low). Check before starting work.
- **`AgentInstructions_scripts/`** — Reusable analysis scripts. Check here before writing a new one. `analyze_logs.py` is the universal log analyzer: `python analyze_logs.py .tasks/task_NNN/logs.json --section all`.
- `implementation_plan.json` validation lives in `_validate_impl_plan` inside `core/phases/planning.py`. This is where most pipeline failures originate.
- `find`/`replace`/`insert_after` fields in `implementation_steps` are for the LLM's own reference — the runtime does **not** parse them, only `code` is validated and used.
- `service` and subtask-level `depends_on` are **dead fields** — not read by any runtime code.
